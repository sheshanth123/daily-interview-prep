# Kantar Data Scientist Interview — Complete Q&A Guide



## Round 1: Core ML & Gen AI Fundamentals

---

### Q1. How does a decision tree algorithm work?

A decision tree is a supervised learning algorithm that makes predictions by learning a hierarchy of if-else rules from data.

**How it works:**
1. Start with the full dataset at the root node.
2. Evaluate every feature and every possible split threshold.
3. Select the split that maximizes **Information Gain** (entropy-based) or minimizes **Gini Impurity**.
4. Partition the data into two (or more) child nodes.
5. Repeat recursively on each child node until a stopping criterion is met:
   - Max depth reached
   - Minimum samples per node
   - Node is pure (all one class)
6. Assign a class label (classification) or mean value (regression) at each leaf node.

**Splitting criteria:**
- **Entropy / Information Gain:** `IG = H(parent) - weighted_avg(H(children))`
- **Gini Impurity:** `G = 1 - Σ p(i)²` — slightly faster, similarly effective.

**Pros:** Interpretable, handles mixed data types, no feature scaling needed.  
**Cons:** Prone to overfitting, unstable (small data changes → very different trees).

---

### Q2. What is pruning and why is it needed?

Without pruning, a decision tree will grow until it perfectly classifies all training samples — memorizing noise and producing very poor generalization on unseen data (overfitting).

**Pre-pruning (Early stopping):** Stop growing the tree before it's fully grown.
- `max_depth`: limits tree depth.
- `min_samples_split`: node must have at least N samples to split.
- `min_samples_leaf`: leaf must have at least N samples.
- `max_features`: consider only a subset of features per split.

**Post-pruning (Cost-Complexity Pruning):** Grow the full tree, then remove branches that don't improve generalization.
- In sklearn: `ccp_alpha` — higher alpha = more aggressive pruning.
- Works by comparing the cost (impurity) of a subtree vs. a single leaf. If the leaf performs comparably, remove the subtree.

**Why it works:** It trades a small increase in training error for a significant reduction in test error by removing low-signal branches.

### Q3. Explain entropy and how it's used in decision trees.

**Entropy** measures the impurity or uncertainty in a dataset:

```
H(S) = -Σ p(x) * log₂(p(x))
```

- Pure node (all one class): H = 0
- Perfectly mixed binary node (50/50): H = 1 (maximum)

**Information Gain** is how much entropy is reduced by a split:

```
IG(S, feature) = H(S) - Σ (|Sv| / |S|) * H(Sv)
```

Where `Sv` is the subset of data for each value of the feature.

The tree picks the feature + threshold with the highest IG — the split that removes the most uncertainty.

**Gini Impurity** (alternative):
```
Gini = 1 - Σ p(i)²
```
Computationally cheaper (no log). Slightly favors larger partitions. In practice, entropy and Gini give very similar trees.

**Example:** A feature that perfectly separates classes achieves IG equal to the parent's entropy — maximum possible gain. A feature that doesn't help achieves IG = 0.

### Q4. What is the bias-variance tradeoff?

**Bias:** Error from incorrect assumptions in the model. A high-bias model is too simple — it underfits and misses patterns in both training and test data (e.g., fitting a line to nonlinear data).

**Variance:** Error from sensitivity to small fluctuations in training data. A high-variance model is too complex — it overfits and captures noise (e.g., a deep unpruned decision tree).

**Decomposition of total error:**
```
Total Error = Bias² + Variance + Irreducible Noise
```

Irreducible noise is the inherent randomness in the data — you can't reduce it with any model.

**The tradeoff:** As you increase model complexity:
- Bias decreases (model fits training data better)
- Variance increases (model becomes more sensitive to training data)

**How to manage it:**
| High Bias | High Variance |
|---|---|
| Add more features | Reduce model complexity |
| Use more complex model | Add regularization (L1/L2) |
| Reduce regularization | Use dropout or pruning |
| Train longer | Get more training data |
| Use ensemble (boosting) | Use ensemble (bagging) |

**Diagnosing:** If training error is high → high bias. If training error is low but test error is high → high variance.

### Q5. What are bagging and boosting, and how do they differ?

Both are ensemble methods that combine multiple weak models to build a stronger one.

**Bagging (Bootstrap Aggregating):**
- Train N models independently on random bootstrap samples (sampling with replacement).
- Average predictions (regression) or majority vote (classification).
- Models run **in parallel**.
- Reduces **variance**.
- Example: **Random Forest** (also randomizes features at each split for extra decorrelation).

**Boosting:**
- Train models **sequentially**. Each model focuses on the errors of the previous one.
- Misclassified samples get higher weights in the next round (AdaBoost) or the next model fits residuals (Gradient Boosting).
- Reduces **bias**.
- Examples: **XGBoost**, **LightGBM**, **AdaBoost**, **CatBoost**.

| | Bagging | Boosting |
|---|---|---|
| Training | Parallel | Sequential |
| Reduces | Variance | Bias |
| Overfitting risk | Low | Higher (especially on noisy data) |
| Speed | Faster | Slower |
| Best for | High-variance models | Weak learners |

**When to use which:** Use Random Forest when you need a robust, low-variance model quickly. Use XGBoost/LightGBM when you need top predictive performance and can tune carefully.

### Q6. Explain dropout in neural networks.

Dropout is a regularization technique that randomly "drops" (sets to zero) a fraction of neuron activations during each forward pass of training.

**Mechanics:**
- During training, each neuron is independently deactivated with probability `p` (e.g., p=0.3 means 30% dropout).
- The remaining activations are scaled up by `1/(1-p)` to maintain expected output magnitude.
- At inference time, all neurons are active (no dropout), but no scaling is needed because the scale was adjusted during training (inverted dropout).

**Why it works:**
1. **Prevents co-adaptation:** Neurons can't rely on specific other neurons always being present. Forces redundant representations.
2. **Implicit ensemble:** With N neurons, there are 2^N possible sub-networks. Dropout approximately trains all of them simultaneously and averages their predictions at inference.
3. **Adds noise:** Acts as data augmentation on activations.

**Best practices:**
- Typical dropout rates: 0.2–0.5 for hidden layers, 0.1–0.2 for input layers.
- Most effective in large, over-parameterized networks.
- Can hurt small networks or when data is already limited.
- Spatial Dropout for CNNs (drops entire feature maps rather than individual activations).

### Q7. What is L1 vs L2 regularization?

Regularization adds a penalty term to the loss function to discourage large weights, reducing overfitting.

**L2 Regularization (Ridge):**
```
Loss = Original_Loss + λ * Σ w²
```
- Penalizes the square of weights.
- Gradient: pushes all weights proportionally toward zero, but never exactly to zero.
- Effect: small weights for all features — distributed shrinkage.
- Best when: all features are potentially relevant, correlated features present.

**L1 Regularization (Lasso):**
```
Loss = Original_Loss + λ * Σ |w|
```
- Penalizes the absolute value of weights.
- Gradient: constant push toward zero regardless of magnitude — can drive weights exactly to zero.
- Effect: sparse weight vector — acts as built-in feature selection.
- Best when: you suspect only a few features matter.

**Elastic Net:**
```
Loss = Original_Loss + λ₁ * Σ|w| + λ₂ * Σw²
```
- Combines both. Handles correlated features better than pure Lasso. Default choice in many production pipelines.

**Intuition:** L2 is like a spring pulling weights toward zero (restoring force proportional to displacement). L1 is like friction — constant force, stops at zero.

**`λ` (regularization strength):** Higher λ = more regularization = simpler model. Tune via cross-validation.

## Round 2: Deep Dive into Gen AI

---

### Q8. How is a language model different from traditional ML?

**Traditional ML:**
- Task-specific: one model per task (churn prediction, fraud detection).
- Requires labeled, structured data.
- Features are manually engineered by domain experts.
- Model size: thousands to millions of parameters.
- Evaluation: fixed metrics on held-out test set.

**Language Models:**
- General-purpose: one model can do translation, summarization, Q&A, code generation via prompting.
- Trained via self-supervised learning on raw text (predict next token — no labels needed).
- Features (representations) are learned automatically via attention.
- Model size: billions to trillions of parameters.
- Evaluation: much harder — open-ended outputs require human evaluation or LLM-as-judge.

**Key architectural difference:** LLMs use the Transformer architecture with self-attention, enabling them to model long-range dependencies across thousands of tokens — something impossible for classical models on text.

**Key training difference:** LLMs use next-token prediction on internet-scale data, allowing emergent capabilities (reasoning, in-context learning) that weren't explicitly trained for.

---

### Q9. What is the role of attention in transformers?

The attention mechanism allows each token in a sequence to dynamically attend to all other tokens, weighting them by relevance.

**Scaled Dot-Product Attention:**
```
Attention(Q, K, V) = softmax(QKᵀ / √d_k) * V
```

- **Q (Query):** What am I looking for?
- **K (Key):** What do I contain?
- **V (Value):** What do I pass forward if selected?

Each token projects its embedding into Q, K, V vectors. The dot product between Q and K gives a similarity score; softmax normalizes it into attention weights; V is the weighted combination passed forward.

**Multi-Head Attention:**
- Runs attention H times in parallel with different learned projections.
- Each head can attend to different relationship types (syntactic, semantic, positional).
- Outputs are concatenated and linearly projected.

**Why it matters:**
- Captures long-range dependencies without vanishing gradients (unlike RNNs).
- Parallel computation over the full sequence (unlike sequential RNNs).
- Enables the model to resolve coreference (e.g., "The cat sat on the mat because **it** was tired" — "it" = cat).

**Self-attention vs Cross-attention:**
- Self-attention: tokens attend to other tokens in the same sequence (encoder/decoder).
- Cross-attention: decoder tokens attend to encoder outputs (used in translation, RAG).

### Q10. What is fine-tuning in LLMs?

Fine-tuning adapts a pre-trained LLM to a specific task or domain by continuing training on a curated dataset.

**Why fine-tune?**
- The base model may lack domain-specific vocabulary or knowledge.
- You need consistent output format or style.
- You want to reduce prompt length (and cost) by "baking in" instructions.
- You need reliable behavior that prompt engineering can't guarantee.

**Types of fine-tuning:**

**1. Full Fine-tuning:** Update all model weights. Expensive in compute and memory. Risk of catastrophic forgetting. Best results but rarely justified.

**2. Instruction Fine-tuning:** Train on (prompt, response) pairs. Teaches the model to follow instructions. This is how ChatGPT-style models are built.

**3. RLHF (Reinforcement Learning from Human Feedback):** Fine-tune using human preference data. Aligns model behavior with human values. Computationally intensive.

**4. LoRA (Low-Rank Adaptation):** Freeze base model weights. Add small trainable low-rank matrices to attention layers. Only trains ~0.1–1% of parameters. Memory-efficient, fast, easily swappable. Industry default for cost-effective fine-tuning.

**5. QLoRA:** LoRA on a quantized (4-bit) base model. Enables fine-tuning 70B models on a single GPU.

**When NOT to fine-tune:** If prompt engineering or RAG solves the problem — fine-tuning is expensive and creates maintenance burden (re-tune when base model updates).

### Q11. Explain RAG and how the retrieval component works.

RAG (Retrieval-Augmented Generation) is an architecture that grounds LLM responses in external, up-to-date knowledge retrieved at inference time.

**Why RAG?**
- LLM knowledge is frozen at training cutoff.
- Private/proprietary data can't be in training data.
- Reduces hallucinations by grounding answers in source documents.
- Cheaper and faster to update than fine-tuning.

**Full RAG Pipeline:**

**Offline (Indexing):**
1. Load documents (PDFs, web pages, databases).
2. Chunk documents into segments (see Q13 for strategies).
3. Embed each chunk using an embedding model (e.g., `text-embedding-3-large`).
4. Store embeddings in a vector database (Pinecone, Weaviate, FAISS, ChromaDB).

**Online (Query time):**
1. User submits a query.
2. Embed the query using the same embedding model.
3. Perform similarity search (cosine similarity) in the vector DB.
4. Retrieve top-k most relevant chunks.
5. Construct a prompt: `[System instructions] + [Retrieved chunks] + [User query]`.
6. LLM generates a response grounded in the retrieved context.

**Advanced RAG techniques:**
- **Reranking:** Use a cross-encoder to rerank top-k retrieved chunks before passing to LLM.
- **HyDE (Hypothetical Document Embeddings):** Generate a hypothetical answer, embed it, use that for retrieval.
- **Query expansion:** Rewrite the query into multiple variants before retrieval.
- **Self-RAG:** LLM decides when to retrieve vs. answer from memory.

### Q12. What are hallucinations in LLMs and how can RAG help?

**What are hallucinations?**
LLMs generate text by predicting the most probable next token — they don't "look up" facts. This means they can produce confident, fluent, but completely fabricated outputs:
- Citing papers that don't exist.
- Inventing statistics or quotes.
- Mixing up facts about similar entities.
- Filling knowledge gaps with plausible-sounding fiction.

**Why do they happen?**
- LLMs optimize for fluency, not factual accuracy.
- Training data is noisy and contains contradictions.
- The model interpolates between seen patterns — novel queries can trigger confabulation.
- Low-frequency facts are more likely to be hallucinated.

**How RAG helps:**
- Grounds responses in specific, retrieved source documents.
- The prompt instructs the model: "Answer ONLY based on the provided context."
- If the answer isn't in the retrieved context, a well-prompted RAG system should say "I don't know" rather than fabricate.
- Enables citation — the model can point to the source chunk.

**Limitations of RAG for hallucinations:**
- The model can still hallucinate *within* the context (misquoting or misinterpreting a retrieved passage).
- If retrieval fails (wrong chunk retrieved), hallucination is still possible.
- Complementary approaches: output verification, confidence calibration, Constitutional AI.

### Q13. Compare chunking methods and when to use each.

Chunking is the process of splitting documents into smaller pieces before embedding. The right strategy significantly impacts retrieval quality.

**1. Fixed-size chunking**
- Split every N tokens/characters regardless of content.
- Pros: Simple, fast, predictable chunk sizes.
- Cons: Breaks sentences mid-thought; loses semantic coherence.
- Use when: Large homogeneous text, speed is priority, content is uniform.

**2. Sentence chunking**
- Split on sentence boundaries using NLP tools (spaCy, NLTK).
- Pros: Semantically coherent units.
- Cons: Variable sizes; very short sentences create noisy chunks.
- Use when: News articles, Q&A datasets, dialogue.

**3. Paragraph / section chunking**
- Split on paragraph breaks or markdown headers.
- Pros: Preserves topical coherence; leverages document structure.
- Cons: Highly variable sizes; very long paragraphs may exceed context limits.
- Use when: Structured documents (reports, wikis, technical docs).

**4. Recursive chunking**
- Tries to split on paragraph → sentence → word in order, keeping chunks under a target size.
- Pros: Balances size and semantic coherence. Handles variable-length documents well.
- Cons: Slightly more complex than fixed-size.
- Use when: General-purpose RAG pipelines. Default in LangChain's `RecursiveCharacterTextSplitter`.

**5. Semantic chunking**
- Group sentences by embedding similarity. Start a new chunk when similarity drops below a threshold.
- Pros: Highest semantic coherence; captures topic shifts naturally.
- Cons: Computationally expensive; requires embedding at index time.
- Use when: Long, complex documents with multiple distinct topics (research papers, legal contracts).

**6. Parent-child / hierarchical chunking**
- Store large "parent" chunks for context, index small "child" chunks for retrieval precision.
- Return parent chunk to LLM after child is retrieved.
- Use when: You need precise retrieval AND rich context.

**Universal best practice:** Always add **chunk overlap** (10–20% of chunk size) so information at boundaries isn't lost between adjacent chunks.

## Round 3: Gen AI Strategy & Trade-offs

---

### Q14. When is it better to use RAG over a base LLM?

**Use RAG when:**
- Knowledge is **private or proprietary** and wasn't in training data (internal documents, CRM data, product catalogues).
- Information **changes frequently** — you can update the vector DB without retraining the model.
- **Factual accuracy with citations** is critical (legal, medical, finance).
- Knowledge base is **too large** to fine-tune on economically.
- You need **transparency** — RAG lets you show users exactly which documents were used.

**Use base LLM (no RAG) when:**
- The task is **generative/creative** (writing, brainstorming) — retrieval adds no value.
- Knowledge is **stable and well-represented** in training data (general world knowledge).
- **Low latency** is critical — RAG adds retrieval overhead (50–200ms+).
- The query is about **reasoning or synthesis**, not recall.

**Use fine-tuning when:**
- You need the model to **reliably produce a specific format** or style.
- Domain-specific **vocabulary or jargon** is not well-represented in the base model.
- You want to **reduce prompt length** at scale (cost savings).

**Trade-off framing for interviews:**
> "While RAG offers up-to-date, citeable responses, it comes at the cost of retrieval latency and pipeline complexity. For a use case like internal knowledge search where accuracy and recency matter, I'd choose RAG. For a creative copywriting assistant, a base LLM or fine-tuned model is sufficient and faster."

### Q15. What are the trade-offs of dense vs sparse retrieval?

**Sparse Retrieval (BM25, TF-IDF):**
- Keyword-based: scores documents by term frequency and inverse document frequency.
- **Pros:** Fast, no training required, interpretable, works well for exact keyword matches, good for specialized terminology.
- **Cons:** Fails on semantic similarity — "myocardial infarction" ≠ "heart attack". Sensitive to vocabulary mismatch.
- Use when: Legal, medical, or technical domains where exact terms matter; when compute budget is limited.

**Dense Retrieval (Bi-encoders: DPR, E5, BGE, OpenAI embeddings + FAISS):**
- Embedding-based: encodes query and documents into vector space, retrieves by cosine similarity.
- **Pros:** Captures semantic meaning, handles paraphrases and synonyms, more robust to vocabulary mismatch.
- **Cons:** Requires training or a pre-trained embedding model, higher compute and storage cost, harder to debug ("why was this chunk retrieved?"), can miss exact keyword matches.
- Use when: Open-domain Q&A, consumer-facing search, queries phrased in natural language.

**Hybrid Retrieval (Best of both worlds):**
- Run both BM25 and dense retrieval independently, then merge results using **Reciprocal Rank Fusion (RRF)** or a linear combination of scores.
- **RRF:** `score(d) = Σ 1 / (k + rank(d))` where k is a constant (typically 60).
- This is the production standard for most serious RAG systems.

| | Sparse (BM25) | Dense | Hybrid |
|---|---|---|---|
| Semantic understanding | Low | High | High |
| Exact keyword match | High | Low | High |
| Setup complexity | Low | Medium | High |
| Latency | Very fast | Fast | Moderate |
| Interpretability | High | Low | Medium |

### Q16. How would you improve chunking strategies in a pipeline?

**Step 1: Diagnose first**

Ask: Are retrieved chunks missing relevant content, or are they noisy/irrelevant?
- Run a sample of 20–50 queries, inspect retrieved chunks manually.
- Compute retrieval metrics: MRR@k, Recall@k, NDCG on a labeled eval set.

**Step 2: Common improvements**

- **Add overlap:** If boundary context is lost, increase chunk overlap from 0 to 10–20%.
- **Adjust chunk size:** Too small = noisy, too large = dilutes relevance signal. Typical sweet spot: 256–512 tokens.
- **Switch to semantic chunking:** For complex documents where topics shift within sections.
- **Use document structure:** Split on headings/sections rather than arbitrary token counts.
- **Add metadata:** Tag each chunk with source, section title, date. Enable **metadata filtering** to pre-filter by date or category before vector search.
- **Parent-child indexing:** Index small chunks (128 tokens) for retrieval precision, return large parent chunk (512 tokens) to LLM for context.
- **Query-aware chunking:** Generate multiple chunk versions at different granularities; retrieve the best match per query.

**Step 3: Evaluate impact**

Always measure before and after using your labeled eval set. Chunking changes can have surprising effects — sometimes simpler is better.

### Q17. When would you fine-tune vs prompt-engineer an LLM?

**Decision framework (in order of cost/effort):**

**1. Zero-shot prompting (free, instant):**
- Just describe the task clearly.
- Works for most standard tasks.

**2. Few-shot prompting (free, fast):**
- Add 2–5 examples in the prompt.
- Dramatically improves formatting and style consistency.
- Use when the model understands the task but isn't formatting correctly.

**3. Prompt optimization / System prompt engineering:**
- Elaborate system prompts with detailed instructions, personas, constraints.
- Use when behavior is complex but describable in language.

**4. RAG (medium cost, infrastructure needed):**
- When the issue is missing knowledge, not behavior.

**5. LoRA / QLoRA fine-tuning (moderate cost):**
- When consistent behavior can't be achieved with prompting.
- When domain-specific vocabulary is outside training distribution.
- When you need to reduce prompt length for cost/latency at scale.
- When you need reliable structured output (JSON with specific schema).

**6. Full fine-tuning (high cost, rarely justified):**
- When you're training from scratch or need to significantly reshape model behavior.
- Typically only for research or very large-scale production systems.

**Rule of thumb:**
> Exhaust prompt engineering first → try RAG if knowledge is the gap → try LoRA if behavior is the gap → full fine-tune only when you have a clear production case that justifies it.

### Q18. Walk me through how you'd build a customer churn prediction system.

**1. Problem Definition**
- Define "churn" precisely with the business: cancel subscription? No purchase in 90 days? Dormant for 30 days?
- Define the prediction horizon: predict churn in the next 30/60/90 days?
- Understand the business action: what happens to predicted churners? (Retention campaign, discount offer, outreach call)
- Align on acceptable false-positive rate — sending retention offers to non-churners has a cost.

**2. Data Collection & Feature Engineering**
- **Behavioral features:** Login frequency, feature usage, time since last activity, session duration trends, support tickets raised.
- **Transactional features:** Purchase recency, frequency, monetary value (RFM model).
- **Demographic features:** Account age, plan type, geography.
- **Engagement signals:** Email open rates, NPS scores, product feedback.
- Handle class imbalance (churn is typically 5–15%): use SMOTE, class weights, or adjust decision threshold.

**3. Modeling**
- **Baseline:** Logistic Regression (interpretable, fast to train, good benchmark).
- **Step up:** XGBoost or LightGBM — typically best for tabular data. Handle missing values, fast, built-in feature importance.
- **Deep learning (optional):** Temporal models (LSTM, Transformer) if sequential behavior patterns are important.
- **Evaluation metrics:** AUC-ROC for overall discrimination. Precision-Recall curve is more informative under class imbalance. Focus on **recall** — missing a churner (false negative) is usually more costly than a false alarm.
- **Explainability:** Use SHAP values to explain individual predictions to business stakeholders.

**4. Deployment**
- Batch scoring pipeline (weekly or daily) — scores all customers, writes results to CRM or data warehouse.
- CI/CD for model retraining (retrain monthly or when drift is detected).
- **Monitoring:** Track feature distribution drift (PSI score), model performance drift (AUC on recent labeled data), and business outcome metrics (churn rate over time).

**5. Feedback Loop**
- A/B test the retention interventions — measure lift vs. control group.
- Use outcomes to generate labeled data for future retraining.

### Q19. How would you collaborate with a cross-functional team on a Gen AI product?

**Phase 1: Scoping & Alignment**
- Run a kick-off with PM, engineering, legal, domain experts, and leadership.
- Define: What is the user problem? What does success look like? What are the guardrails?
- Translate business goals into measurable ML metrics: "improve user satisfaction" → reduce hallucination rate below 5%, response latency under 2s.
- Identify data sources, access constraints, and compliance requirements early.

**Phase 2: Build & Iterate**
- Build a minimal prototype fast — a working demo beats a long spec document for alignment.
- Maintain a **shared evaluation set** that all stakeholders agree on before building. This prevents "it feels worse" debates later.
- Communicate model limitations proactively: hallucination risk, edge cases, failure modes. Don't let stakeholders discover limitations in production.
- Work with engineers on: latency budgets (affects model choice), infrastructure constraints, API rate limits.
- Loop in legal/compliance on: data usage agreements, output liability, PII handling, model cards.

**Phase 3: Launch & Monitor**
- Define rollout strategy: internal testing → beta users → full rollout.
- Set up monitoring dashboards: usage, latency, error rates, feedback signals.
- Establish a process for handling edge cases reported by users.
- Schedule regular reviews with stakeholders — Gen AI products degrade silently as the world changes.

**Key collaboration principles:**
- Translate technical concepts into business terms. "Hallucination rate" → "how often does it give wrong answers."
- Say "I don't know" early rather than overpromise.
- Build trust by sharing progress, not just outcomes.

### Q20. Describe a time you faced ambiguity in a project. How did you handle it?

Use the **STAR method** to structure your answer. Here is a template with strong signal:

**Situation:** Set the context. What was the project? What specifically was ambiguous?
> e.g., "We were asked to build a recommendation system, but the success metric wasn't defined — different stakeholders had different ideas about what 'good recommendations' meant."

**Task:** What was your responsibility in this situation?
> e.g., "As the lead data scientist, I was responsible for scoping the project and aligning the team before building anything."

**Action:** What concrete steps did you take to resolve the ambiguity?
> Strong signals:
> - Proactively scheduled stakeholder interviews to surface conflicting assumptions.
> - Defined a proxy metric that all parties could agree on.
> - Built a quick prototype to make the problem concrete and facilitate discussion.
> - Documented assumptions explicitly and got sign-off before proceeding.
> - Proposed a phased approach: first validate the simplest version, then iterate.

**Result:** What was the outcome? Quantify where possible.
> e.g., "We aligned on click-through rate as the primary metric, launched in 6 weeks, and saw a 12% increase in engagement in the first month."

**What interviewers look for:**
- Did you seek clarity proactively rather than waiting for direction?
- Did you make a defensible, documented decision when clarity wasn't possible?
- Did you communicate your assumptions clearly to stakeholders?
- Did you adapt when new information emerged?

**Common mistake:** Saying "I just figured it out" without explaining the process. Show the thinking, not just the outcome.
